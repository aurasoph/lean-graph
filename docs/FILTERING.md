# Filtering Design

## Core Principle

Every node in the graph corresponds to a declaration that has a **standalone entry in the Mathlib documentation** (doc-gen4). A declaration is included if and only if it would appear as its own page in the docs.

This means the graph and the docs are in 1:1 correspondence: if you can look it up in Mathlib docs, it is a node. If it doesn't have its own docs page, it is not a node.

---

## The Filter

A declaration is included when all of these hold:

1. **Has a source range** (`isExplicitAPI`) — it was written by a human, not generated by the compiler. This excludes equation lemmas, match blocks, auto-generated constructors that share their parent's source position, and other elaborator output.

2. **Not an auxiliary recursor** — `casesOn`, `recOn`, `brecOn` siblings generated from inductive definitions.

3. **Not a noConfusion lemma** — auto-generated injectivity/disjointness lemmas for constructors.

4. **Not a projection function** — field accessors like `Mul.mul`, `Norm.norm`, `CategoryStruct.comp`. These are shown as fields on the parent structure's documentation page, not as standalone entries.

Everything else is included: typeclass instances, tactic-layer declarations, opaque definitions, axioms, structures, classes.

Pass `--include-aux` to bypass all filtering (exhaustive/debug mode).

---

## What This Matches

This filter is designed to mirror what doc-gen4 calls "rendered" declarations — the ones that get their own `#doc-gen4-anchor` entry. The match is intentional: the graph should make the same inclusion decisions as the documentation generator.

The key insight is that doc-gen4's blacklist (`isBlackListed`) and the `render` field together decide exactly which declarations appear as standalone pages, and our filter reproduces that logic:

- `isExplicitAPI` ↔ doc-gen4's source-range check (no range = compiler-generated)
- `isAuxRecursor` ↔ doc-gen4's recursor blacklist
- `isNoConfusion` ↔ doc-gen4's noConfusion blacklist
- `isProjectionFn` ↔ doc-gen4's `render = false` for projection functions

---

## Expansion in Proof Dependencies

When building proof dependency edges, the filter is applied with a DFS expansion step. If a declaration in the proof term is excluded (e.g. an equation lemma or match block), we look inside its body for real mathematical content, because compiler-generated wrappers often contain genuine dependencies inside them.

Projection functions are **not** expanded through — their bodies are pure field accesses (`fun inst => inst.fieldN`) with no mathematical content to surface.

---

## Edge Types

The unified graph records six kinds of dependency:

| Kind | Description |
|------|-------------|
| `extends` | Structure/class inheritance (`A extends B`) |
| `field` | Composition via field/parameter (`A has a field of type B`) |
| `sig` | Type appearing in a declaration's signature |
| `proof` | Theorem invoking another theorem/lemma in its proof body |
| `def` | Definition invoking another declaration in its body |
| `docref` | Backtick reference (`` `Name ``) in a docstring |

`docref` edges are textual — they record which declarations are mentioned by name in documentation. Doc-gen4 turns these same references into hyperlinks; the graph turns them into edges.
